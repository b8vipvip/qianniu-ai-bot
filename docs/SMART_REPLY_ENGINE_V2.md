# Smart Reply Engine V2

## Phase 1：会话状态 + 上下文问题还原 + 本地混合检索

本阶段解决的问题：买家经常只发送“这个呢”“为什么不行”“可以”“还是刚才那个”等依赖上下文的短句。如果直接拿原话检索知识库，即使 Smart Reply Router 不再机械直答，也可能召回错误知识。

### Conversation State

每次回复前从同一客服、同一买家的最近有效对话中本地提取：

- 当前主题 `CurrentTopic`
- 当前商品/服务对象 `CurrentEntity`
- 买家当前意图 `BuyerGoal`
- 上一轮客服待确认问题 `PendingQuestion`
- 当前会话阶段 `ConversationStage`
- 已确认信息 `ConfirmedFacts`

这些状态只用于理解上下文，不会被当成新的店铺事实。

### Contextual Query Rewrite

当上下文依赖度达到阈值时，本地还原用于知识检索的完整问题，不额外调用 AI。

```text
客服：请确认电视上有没有安装酷狗音乐？
买家：有
```

检索输入不再只是“有”，而会补充成类似：

```text
酷狗音乐；请确认电视上有没有安装酷狗音乐？；买家回答/追问：有
```

### Phase 1 Hybrid Retrieval

候选知识分数同时考虑：

- 当前原始问题文本相关度
- 上下文还原后的完整问题相关度
- 知识关键词命中
- 当前会话实体匹配
- 最近上下文主题匹配
- 意图一致性

召回后保持：

```text
全部启用知识
→ 本地混合打分
→ Top 10 候选池
→ Top 3 进入最终提示词
```

被 Query Rewrite 过的问题禁止走 `DIRECT_KNOWLEDGE` 固定答案路径，必须结合上下文生成。

## Phase 2：可选 Embedding 语义检索

统一 API 控制面提供 OpenAI 兼容 `/v1/embeddings`。Bot 可以配置可选语义模型，并把向量语义相似度加入本地混合检索。

- 知识向量持久化到 `%LocalAppData%\QianniuAiBot\data\knowledge-embeddings.json`；
- 查询和缺失的高优先级知识可批量生成向量；
- 同步语义检索等待被限制在约 2.2 秒；
- 任何超时、接口错误或未配置模型都会自动回退 Phase 1 本地检索；
- 纯语义命中不能触发 `DIRECT_KNOWLEDGE`，必须保留足够强的原始文本证据。

## Phase 3：知识策略与可靠度学习

每条知识可以设置：

- `auto` 自动判断；
- `direct` 优先直答；
- `contextual` 必须结合上下文；
- `constraint` 仅作为事实约束；
- 意图、业务实体、适用条件、禁止适用条件和必需上下文。

系统还会根据真实客服行为更新可靠度：

- 直接/上下文路由使用次数；
- 人工确认；
- 客服修改；
- Bot 撤回后人工纠正。

频繁被修改或撤回的知识会失去直接发送资格；经人工反复确认的知识可逐步提高可信度。

## Phase 4：发送前 Answer Validator

AI 或上下文改写答案在真正显示和发送前，会先经过本地快速校验。

检查范围包括：

- 是否真正回答价格、时效、是否支持、操作方法或故障排查等当前意图；
- 是否包含“作为 AI”“系统提示词”“知识库”等机器化或内部措辞；
- 是否出现店铺提示词或知识中没有依据的具体价格、数字或时效；
- 是否擅自声称已退款、已发货、已到账、审核通过等真实订单/售后状态；
- 是否出现百分百、保证、永久、无条件退款等无依据绝对承诺；
- 是否与高可信知识或 `constraint` 事实约束的支持/不支持结论冲突；
- 是否过短、过于笼统或异常过长。

校验结果分三种：

```text
PASS
→ 正常进入去重、稳定性检查和真实发送

REGENERATE
→ 带着具体问题、店铺事实、最相关知识和最近会话只重答一次
→ 修正答案再次校验

MANUAL
→ 阻止自动发送，在 Bot 面板显示错误原因，等待人工处理
```

为了控制延迟和费用：

- 只有本地校验发现可修复问题时才增加一次 AI 调用；
- 第二次仍不合格就直接停止，不循环重试；
- Smart Reply Router 已批准的本地知识原文不会再次调用 AI 校验重答；
- 重复答案修正结果也必须通过同一个校验器。

## Phase 5：自适应回复等待时间

连续消息聚合不再只使用全局固定等待时间，而是继续保留消息类型基础规则，再叠加同一客服、同一买家的真实输入节奏。

学习规则：

- 只记录同一轮连续消息之间 `120–4500ms` 的有效间隔；
- 过滤详细消息重复回放、补抓事件和已经属于下一轮咨询的长间隔；
- 每个买家只保留最近 24 个有效间隔，避免很久以前的输入习惯永久影响当前行为；
- 至少积累 3 个样本后才开始调整；
- 使用最近间隔的第 75 百分位再增加少量缓冲，比简单平均值更不容易被极端间隔带偏。

不同消息类型仍有独立安全边界：

```text
完整句子：300–950ms
未结束短句：650–1550ms
问候语：700–1450ms
图片/媒体：600–1400ms
高密度连续消息：300–720ms
```

原有保护继续保留：

- 单轮累计达到 4 秒后只再等待约 80ms，禁止无限等待；
- 买家发来新消息会取消当前静默计时并重新计算；
- 学习结果只影响该客服与该买家，不会串到其他店铺或其他买家；
- 数据保存在 `%LocalAppData%\QianniuAiBot\data\adaptive-reply-timing.json`；
- 30 天未更新的画像自动清理，最多保留 2000 个；
- 每 30 秒或累计 20 次更新自动保存，程序退出前再次强制保存。

## 后续阶段

### Phase 6：回复质量中心

汇总路由使用率、校验通过率、重答率、人工阻止率、客服纠正率、平均首字和完整回复延迟，为后续 Router 权重和模型路由优化提供真实指标。
