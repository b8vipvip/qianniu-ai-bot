# Smart Reply Engine V2

## Phase 1：会话状态 + 上下文问题还原 + 本地混合检索

本阶段解决的问题：买家经常只发送“这个呢”“为什么不行”“可以”“还是刚才那个”等依赖上下文的短句。如果直接拿原话检索知识库，即使 Smart Reply Router 不再机械直答，也可能召回错误知识。

### Conversation State

每次回复前从同一客服、同一买家的最近有效对话中本地提取：

- 当前主题 `CurrentTopic`
- 当前商品/服务对象 `CurrentEntity`
- 买家当前意图 `BuyerGoal`
- 上一轮客服待确认问题 `PendingQuestion`
- 当前会话阶段 `ConversationStage`
- 已确认信息 `ConfirmedFacts`

这些状态只用于理解上下文，不会被当成新的店铺事实。

### Contextual Query Rewrite

当上下文依赖度达到阈值时，本地还原用于知识检索的完整问题，不额外调用 AI。

示例：

```text
客服：请确认电视上有没有安装酷狗音乐？
买家：有
```

检索输入不再只是：

```text
有
```

而会补充成类似：

```text
酷狗音乐；请确认电视上有没有安装酷狗音乐？；买家回答/追问：有
```

又例如：

```text
前文：正在咨询电视端会员
买家：那手机呢？
```

会把最近主题和当前实体补充进检索查询。

### Phase 1 Hybrid Retrieval

候选知识分数现在同时考虑：

- 当前原始问题文本相关度
- 上下文还原后的完整问题相关度
- 知识关键词命中
- 当前会话实体匹配
- 最近上下文主题匹配
- 意图一致性

仍然只在本地完成，不增加额外 AI 请求。

召回后仍保持：

```text
全部启用知识
→ 本地混合打分
→ Top 10 候选池
→ Top 3 进入最终提示词
```

被 Query Rewrite 过的问题禁止走 `DIRECT_KNOWLEDGE` 固定答案路径，必须结合上下文生成。

## 后续阶段

### Phase 2：Embedding 语义检索

在不改变 Router 接口的前提下，为知识条目维护语义向量，把向量相似度加入现有混合检索分数。知识向量只在新增或修改知识时更新；买家请求只生成当前 Query 向量。

### Phase 3：知识可靠度学习

根据 Bot 回复后的真实客服行为学习每条知识的可靠度：

- 是否被保留
- 是否被客服撤回
- 是否被客服修改
- 是否需要人工补充

可靠度高的知识更容易进入 `DIRECT_KNOWLEDGE`，经常被纠正的知识强制进入上下文生成或人工确认。

### Phase 4：发送前 Answer Validator

在真正发送前增加本地快速校验，并仅在高风险时触发额外 AI 复核，检查知识冲突、店铺规则冲突、错误承诺、重复答案和机器化措辞。
